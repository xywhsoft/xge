# XGE 场景系统说明

本文档固化 XGE V2 场景系统的公开语义、当前实现状态和后续修正方向。XGE 使用现代 lifecycle scene model，不兼容 XGE V1 的旧消息回调模型。

## 设计目标

场景系统用于组织游戏主流程、子流程、暂停菜单、过场、编辑器面板和 RMGUI/APP 工作流。它必须保持轻量，不引入 node/entity/object 系统；实体系统后续作为独立扩展设计。

场景系统目标：

- 明确生命周期：enter、leave、pause、resume、event、update、draw、free。
- 支持 scene stack，用于主场景 hold 住后进入子场景。
- 支持 variable update 和 fixed update。
- 支持一次 draw 前多次 update。
- 与输入、渲染、资源异步加载、XUI 桥接协同。
- 保持 C API 和透明结构，用户可以直接管理 scene 对象生命周期。

## 公开结构

当前公开结构：

```c
typedef struct xge_scene_t xge_scene_t;
typedef xge_scene_t* xge_scene;
typedef int (*xge_scene_lifecycle_proc)(xge_scene pScene);
typedef int (*xge_scene_event_proc)(xge_scene pScene, const xge_event_t* pEvent);
typedef int (*xge_scene_update_proc)(xge_scene pScene, float fDelta);
typedef int (*xge_scene_draw_proc)(xge_scene pScene);

struct xge_scene_t {
	void* pUser;
	xge_scene_lifecycle_proc onEnter;
	xge_scene_lifecycle_proc onLeave;
	xge_scene_lifecycle_proc onPause;
	xge_scene_lifecycle_proc onResume;
	xge_scene_event_proc onEvent;
	xge_scene_update_proc onUpdate;
	xge_scene_draw_proc onDraw;
	xge_scene_lifecycle_proc onFree;
	int bEntered;
	int bPaused;
};
```

约定：

- `pUser` 由用户持有，XGE 不解释其内容。
- 回调返回 `0` 表示成功，非零值会向上传播并可终止当前流程。
- `bEntered` 和 `bPaused` 是可读状态，但用户不应直接写入。
- 场景对象可以是静态对象、栈外对象、xrt allocator 分配对象或用户自定义生命周期对象。
- XGE 调用 `onFree` 只是生命周期通知，不强制释放场景内存；是否释放由用户回调决定。

## 栈语义

当前公开 API：

```c
int xgeSceneSet(xge_scene pScene);
int xgeScenePush(xge_scene pScene);
int xgeScenePop(void);
int xgeSceneReplace(xge_scene pScene);
xge_scene xgeSceneCurrent(void);
int xgeSceneCount(void);
int xgeSceneDispatchEvent(const xge_event_t* pEvent);
```

语义：

- `xgeSceneSet`：清空现有场景栈，进入新场景。
- `xgeScenePush`：暂停当前场景，压入并进入新场景。
- `xgeScenePop`：离开当前场景，恢复上一个场景。
- `xgeSceneReplace`：离开当前栈顶场景，用新场景替换并进入。
- `xgeSceneCurrent`：返回栈顶场景。
- `xgeSceneCount`：返回当前栈深度。
- `xgeSceneDispatchEvent`：把事件派发给当前栈顶场景。

栈深度由 `XGE_SCENE_STACK_MAX` 限制。第一阶段默认为 16，足够覆盖主场景、弹窗、暂停菜单、临时子流程等常见用法。

## 生命周期顺序

### Set

`xgeSceneSet(A)`：

1. 从栈顶到栈底依次对旧场景调用 `onLeave`。
2. 对旧场景调用 `onFree`。
3. 清空栈。
4. 压入 A。
5. 调用 A 的 `onEnter`。

### Push

当前栈顶为 A，调用 `xgeScenePush(B)`：

1. 调用 A 的 `onPause`。
2. 压入 B。
3. 调用 B 的 `onEnter`。

典型用途：

- 暂停菜单。
- 子关卡。
- 模态 UI。
- 战斗结算。
- 临时编辑器面板。

### Pop

当前栈为 A, B，调用 `xgeScenePop()`：

1. 调用 B 的 `onLeave`。
2. 调用 B 的 `onFree`。
3. 弹出 B。
4. 调用 A 的 `onResume`。

### Replace

当前栈顶为 A，调用 `xgeSceneReplace(B)`：

1. 调用 A 的 `onLeave`。
2. 调用 A 的 `onFree`。
3. 用 B 替换栈顶。
4. 调用 B 的 `onEnter`。

## Update 策略

当前公开 API：

```c
int xgeSceneUpdateStrategySet(int iMode, float fFixedStep, int iMaxUpdates);
void xgeSceneUpdateStrategyGet(int* pMode, float* pFixedStep, int* pMaxUpdates);
```

支持模式：

- `XGE_UPDATE_VARIABLE`：每帧调用一次 `onUpdate(scene, delta)`。
- `XGE_UPDATE_FIXED`：按固定步长累积，可在一次 draw 前执行多次 update。

默认值：

- mode：`XGE_UPDATE_VARIABLE`
- fixed step：`1.0f / 60.0f`
- max updates：`1`

固定更新策略：

- 每帧把真实 `delta` 加入 accumulator。
- accumulator 大于 fixed step 时执行 update。
- 单帧最多执行 `iMaxUpdates` 次。
- 如果达到最大次数仍追不上，清空 accumulator，避免死亡螺旋。

建议：

- 动作、物理、确定性逻辑使用 fixed update。
- UI、编辑器、普通小游戏可以使用 variable update。
- 不要在 `onDraw` 中修改核心游戏状态。

## Draw 与渲染生命周期

场景 `onDraw` 是游戏循环模式下的主要绘制入口。

约定目标：

- 自动游戏循环会在场景绘制阶段包裹 begin/end。
- 用户通常只在 `onDraw` 中调用 `xgeDraw*`、`xgeShape*`、`xgeXuiPaint` 等绘制 API。
- APP/RMGUI 手动刷新模式可以不依赖场景系统，直接使用 `xgeBegin/xgeEnd/xgePresent`。

当前状态：

- `xgeBegin/xgeEnd/xgeFlush/xgePresent` 已存在。
- 自动游戏循环已经能调用场景 update/draw。
- 后续 command queue 完成后，需要进一步统一 scene draw 与 render command flush 的关系。

## 事件派发

事件通过 `xgeSceneDispatchEvent` 进入当前场景。

规则：

- 默认只派发给栈顶场景。
- 被 pause 的下层场景不接收输入。
- 如果需要让下层场景继续收到部分事件，应由栈顶场景主动转发，或后续增加显式 flag。

典型集成：

- 平台层把键盘、鼠标、触摸、gamepad、text input、IME 事件转换成 `xge_event_t`。
- XGE 先更新全局输入状态。
- 再派发给当前 scene。
- scene 可继续把事件交给 XUI。

建议顺序：

```c
int on_event(xge_scene pScene, const xge_event_t* pEvent)
{
	my_scene_t* pSelf = (my_scene_t*)pScene->pUser;
	if ( xgeXuiDispatchEvent(&pSelf->tXui, pEvent) == XGE_XUI_EVENT_CONSUMED ) {
		return XGE_OK;
	}
	return my_game_handle_event(pSelf, pEvent);
}
```

## XUI 集成

XUI 不属于 XGE 场景系统，但经常由 scene 持有。

推荐结构：

```c
typedef struct my_scene_t {
	xge_scene_t tScene;
	xge_xui_context_t tXui;
} my_scene_t;
```

推荐流程：

- `onEnter` 初始化资源和 XUI tree。
- `onEvent` 先交给 XUI，再处理游戏输入。
- `onUpdate` 调用 `xgeXuiUpdate` 和游戏逻辑。
- `onDraw` 绘制游戏内容，再调用 `xgeXuiPaint`。
- `onLeave` 停止场景内音频、取消异步加载或断开回调。
- `onFree` 释放 XUI、资源引用和用户内存。

## 异步资源与场景

场景经常触发异步资源加载。约定：

- async callback 不应直接销毁或切换 scene。
- 场景离开时要取消或忽略未完成请求。
- 资源对象使用引用计数，回调中取到资源后再交给 scene 持有。
- fallback texture/font/sound 可用于微端和流式加载。

建议用户在 scene userdata 内保存 generation/token：

```c
typedef struct my_scene_t {
	int iAliveToken;
	xge_texture_t tFallback;
	xge_texture_t tRealTexture;
} my_scene_t;
```

回调只在 token 仍有效时提交资源，避免场景已离开后写入失效对象。

## 回调中切换场景

目标设计要求：场景切换不应在 lifecycle/update/draw/event 回调中立即修改栈，而应记录 pending operation，在安全点处理。

安全点：

- frame start
- update 前
- draw 后

当前实现仍以直接修改栈为主。后续修正任务：

- 增加 pending scene op。
- 在回调嵌套期间禁止直接修改栈。
- 在安全点统一执行 pending op。
- 为回调中 push/pop/replace 增加测试。

在该修正完成前，用户应避免在 `onDraw` 中直接切换场景；推荐在 `onUpdate` 或 `onEvent` 中设置用户自己的 pending flag，并在下一次 update 开头调用 XGE scene API。

## 线程约束

第一阶段约定：

- 场景栈操作在主线程或游戏线程调用。
- scene 回调不应从后台加载线程直接触发。
- 后台线程可以完成 IO 和解码，再通过 XGE async/GPU upload queue 把结果交回主线程或 render thread。

后续 render thread 引入后：

- scene update 仍在 game thread。
- render command submission 可从 game thread 进入队列。
- GL resource creation/destroy 在 render thread。

## 错误处理

场景 API 返回 `int` 枚举错误码。

常见错误：

- 未初始化时调用 scene API：`XGE_ERROR_NOT_INITIALIZED`。
- 传入 NULL 场景：按 API 语义处理，`Set(NULL)` 可用于清空，`Push(NULL)` 应失败。
- 栈满：返回错误。
- 回调返回非零：向上传播。

如果 `onEnter` 失败：

- 当前 API 应返回错误。
- 后续需要确保失败场景不会留下半进入状态。
