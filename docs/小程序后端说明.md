# XGE 小程序后端说明

小程序是 XGE V2 第一版核心目标，但它不走 Sokol。XGE 需要独立小程序平台后端和 JS bridge，以适配小程序运行时的 Canvas、WebGL2、触摸、资源加载、音频和生命周期限制。

## 目标

第一版小程序后端目标：

- 创建 WebGL2 Canvas context。
- 驱动 XGE game loop。
- 转发 touch 输入。
- 在平台支持时转发 keyboard/text input。
- 提供资源加载 bridge。
- 提供基础音频策略。
- 支持 XUI/RMGUI 的手动刷新和 dirty rect 工作流。
- 与 GL-family 渲染后端共用 GLES3/WebGL2 映射层。

非目标：

- 不依赖 Sokol。
- 不要求第一版支持小程序多线程。
- 不要求第一版支持所有小程序平台的完整 API 差异。
- 不在 C 层直接访问平台私有 JS 对象。
- 不把小程序特殊逻辑散落到 XGE core。

## 总体结构

建议文件结构：

```text
src/xge_platform_miniprogram.c
src/xge_platform_miniprogram.h
platform/miniprogram/xge_miniprogram.js
platform/miniprogram/xge_miniprogram_types.d.ts
examples/miniprogram_hello/
build_miniprogram.bat
build_miniprogram.sh
```

模块职责：

- C 后端负责 XGE 平台抽象、事件结构、资源请求、frame hook。
- JS bridge 负责创建 Canvas/WebGL2、转发事件、读取文件、调度 requestAnimationFrame。
- 渲染后端仍走 XGE GL-family 层，并通过 WebGL2 限制集运行。

## 初始化流程

推荐初始化流程：

```text
JS app/page init
  -> create canvas
  -> get WebGL2 context
  -> load wasm/native module if available
  -> call xgeMiniProgramAttachCanvas
  -> xgeInit
  -> xgeSceneSet or user init
  -> start game loop
```

C-facing hooks 概念：

```c
typedef struct xge_miniprogram_desc_t {
	void* pCanvas;
	void* pWebGLContext;
	int iWidth;
	int iHeight;
	float fDevicePixelRatio;
	void* pUser;
} xge_miniprogram_desc_t;

int xgeMiniProgramInit(const xge_miniprogram_desc_t* pDesc);
void xgeMiniProgramUnit(void);
int xgeMiniProgramFrame(double fTimeSeconds);
int xgeMiniProgramResize(int iWidth, int iHeight, float fDevicePixelRatio);
```

当前已固化的第一版 C API：

```c
int xgeMiniProgramInit(const xge_miniprogram_desc_t* pDesc);
int xgeMiniProgramInitSimple(int iWidth, int iHeight, float fDevicePixelRatio);
void xgeMiniProgramUnit(void);
int xgeMiniProgramSetBridge(const xge_miniprogram_bridge_t* pBridge);
int xgeMiniProgramFrame(double fTimeSeconds);
int xgeMiniProgramResize(int iWidth, int iHeight, float fDevicePixelRatio);
int xgeMiniProgramTouch(int iPhase, const xge_miniprogram_touch_t* pTouches, int iCount);
int xgeMiniProgramTouchOne(int iPhase, int iId, float fX, float fY, float fForce);
int xgeMiniProgramText(uint32_t iCodepoint);
int xgeMiniProgramRequestFrame(void);
```

约束：

- `xgeMiniProgramInit` 会把平台后端标记为 `XGE_PLATFORM_BACKEND_MINIPROGRAM`。
- 图形后端标记为 `XGE_GPU_BACKEND_WEBGL2`。
- C 层不直接保存 JS 对象，只保存 `void*` 透传句柄。
- `xgeMiniProgramInitSimple` 和 `xgeMiniProgramTouchOne` 是 WASM/JS 优先入口，JS bridge 不直接打包 C 结构体，避免指针宽度、结构体对齐和 `uint64_t` 传参差异。
- 结构体版本保留给 C/C++ 嵌入层、本地测试或未来更低层的原生 bridge 使用。
- resource bridge 通过 `xge_miniprogram_bridge_t` 接入，并注册为 `res://` provider。

## Canvas 与 WebGL2

小程序后端基线为 WebGL2 Canvas。

约定：

- 优先请求 WebGL2 context。
- 如果平台只支持 WebGL1，第一版可以直接报不支持。
- Canvas 逻辑尺寸和实际像素尺寸需要区分。
- `devicePixelRatio` 变化时，必须触发 resize。
- WebGL context lost/restored 需要作为后续任务纳入。

WebGL2 限制：

- Shader 使用 GLSL ES 3.00。
- 需要 precision 声明。
- VAO、FBO、texture format 必须按 WebGL2 可用集合约束。
- 不依赖桌面 GL 扩展。
- 不使用 client-side vertex array。
- 不使用不兼容 WebGL2 的同步/读写模式。

## 渲染后端关系

小程序不是独立渲染 API，而是平台层和 WebGL2 context 提供者。

关系：

```text
miniprogram JS bridge
  -> Canvas/WebGL2 context
  -> xge_platform_miniprogram
  -> XGE GL-family backend
  -> command queue / renderer
```

XGE 的渲染 API 不应感知当前是否小程序，只通过 platform/gpu backend caps 判断能力。

必须验证：

- sprite draw。
- shape draw。
- text/XRF draw。
- render target。
- shader/material。
- XUI paint。
- premultiplied alpha。

## Game Loop

小程序 game loop 由 JS bridge 调度。

推荐：

```text
requestAnimationFrame
  -> xgeMiniProgramFrame(time)
  -> xgeFrame()
  -> xgeFlush()
```

手动刷新/RMGUI 模式：

- XUI 或应用请求 refresh 时，JS bridge 安排下一帧。
- 没有 dirty 状态时可以停止连续 requestAnimationFrame。
- 游戏模式默认持续帧循环。

需要区分：

- `xgeFrame()`：更新游戏和场景。
- `xgeBegin/xgeEnd/xgePresent()`：APP/RMGUI 手动绘制。
- 小程序里 present 可能只是 flush 到 WebGL backbuffer，真正显示由平台帧结束控制。

## 输入事件

第一版必须支持 touch。

事件映射：

- touchstart -> `XGE_EVENT_TOUCH_BEGIN`
- touchmove -> `XGE_EVENT_TOUCH_MOVE`
- touchend -> `XGE_EVENT_TOUCH_END`
- touchcancel -> `XGE_EVENT_TOUCH_CANCEL`

坐标规则：

- JS 传入 CSS/逻辑坐标和 devicePixelRatio。
- C 侧统一转换为 XGE 当前像素坐标或逻辑坐标。
- 多点触控保留 touch id。

建议 JS event payload：

```js
{
	type: "touchstart",
	time: 123.456,
	touches: [
		{ id: 1, x: 120, y: 80, force: 1.0 }
	],
	changedTouches: [
		{ id: 1, x: 120, y: 80, force: 1.0 }
	]
}
```

Keyboard/text input：

- 小程序平台能力差异较大。
- 第一版只在平台支持时启用。
- Text input 和 IME 依赖系统输入控件或平台 API，不实现内建 IME。
- XUI input 控件需要能接收 composition/update/end 事件，但候选窗由系统处理。

## 资源加载

小程序资源加载不应走桌面文件系统假设。

资源来源：

- 小程序包内静态资源。
- 用户数据目录缓存。
- 网络下载后缓存。
- xpack 包。
- JS bridge 提供的 ArrayBuffer。

XGE 层统一使用资源 provider：

```text
file path / res:// path
  -> xgeResourceLoad
  -> miniprogram provider
  -> JS loadFile / readPackage / readCache
  -> ArrayBuffer
  -> C memory view
```

约定：

- 小程序后端必须注册 `res://` provider 或平台 provider。
- 同步加载可以用于小资源和启动资源。
- 异步加载用于图片、音频、字体、xpack。
- 大资源应支持 fallback texture/font/sound。

需要注意：

- JS ArrayBuffer 生命周期必须明确，C 侧不能持有已释放或移动的内存。
- 如果通过 WASM heap 复制，必须由 XGE 释放对应内存。
- 如果平台允许文件缓存，资源 provider 应把缓存路径和版本交给上层或 xrt。

## 音频策略

当前 XGE 音频后端为 miniaudio，但小程序环境是否能直接运行取决于编译形态和平台能力。

第一版需要做两条策略：

1. 验证 miniaudio 在目标小程序 WASM/JS 环境中的可行性。
2. 若不可行，提供 JS audio bridge。

JS audio bridge 目标：

- sound/music/stream 三类语义尽量保持。
- 支持 play/stop/pause/resume。
- 支持 loop。
- 支持 volume group。
- 支持基础 fade。
- 3D 坐标播放第一版可退化为音量/pan 或标记不支持。

音频能力必须通过 caps 暴露，避免用户以为小程序与桌面能力完全一致。

## 构建产物

小程序后端构建产物建议：

```text
dist/miniprogram/
  xge.js
  xge.wasm
  xge_miniprogram.js
  assets/
  project.config.json or platform-specific config
```

第一版构建脚本：

- `build_miniprogram.bat`
- `build_miniprogram.sh`

脚本目标：

- 编译 XGE 到小程序可加载形态。
- 复制 JS bridge。
- 复制示例资源。
- 输出 hello 示例目录。

如果目标平台暂时不能自动编译，脚本可以先生成目录骨架和说明，但 spec 不能标记实现完成。

## 能力报告

小程序后端初始化后应提供平台能力报告：

- WebGL2 是否可用。
- 最大 texture size。
- 支持的 texture format。
- render target 支持情况。
- audio backend 类型。
- touch 支持。
- keyboard/text 支持。
- resource provider 类型。
- 是否支持 WASM。
- 是否支持多线程。

这些能力在 debug 模式可进入 caps dump。

## 与 XUI 的关系

小程序是 XUI/RMGUI 的重要目标环境。

需要保证：

- touch 事件能进入 XUI dispatch。
- text input 能在平台支持时进入 XUI input。
- dirty rect/refresh request 能驱动按需刷新。
- XUI paint 能映射到 WebGL2 绘制。
- XRF 字体可快速加载，减少启动期 TTF rasterize。

APP/RMGUI 模式下，建议使用按需刷新而不是持续 game loop，以节省移动设备电量。

## 风险与限制

主要风险：

- 小程序平台差异大，WebGL2 和 WASM 支持细节可能不同。
- 音频不一定能直接复用 miniaudio。
- 文件系统和缓存策略受平台限制。
- 多线程能力通常受限。
- WebGL2 shader 与桌面 GL shader 差异明显。
- 调试工具和性能分析能力弱于桌面。

处理原则：

- 后端接口集中，不污染 XGE core。
- 能力通过 caps 明确暴露。
- 第一版优先跑通 hello、sprite、text、touch、resource。
- 复杂能力先降级，再逐步补齐。

## 第一阶段任务

spec 中的小程序实现任务保持未完成，直到代码和示例实际落地。

任务顺序：

1. 定义 `XGE_PLATFORM_MINIPROGRAM` 和 `XGE_GPU_WEBGL2` 编译路径。
2. 添加 C-facing platform hook 头文件。
3. 添加 JS bridge 骨架。
4. 添加 WebGL2 context attach 流程。
5. 添加 requestAnimationFrame game loop。
6. 添加 touch event bridge。
7. 添加 resource provider bridge。
8. 验证 miniaudio 或实现 JS audio bridge。
9. 添加 `build_miniprogram` 脚本。
10. 添加 `examples/miniprogram_hello`。
11. 在真实小程序环境做人机冒烟测试。
