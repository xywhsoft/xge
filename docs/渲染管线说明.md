# XGE 渲染管线说明

本文档固化 XGE V2 第一阶段渲染管线的实现边界。它用于约束后续开发：公开 API 保持轻量 C 风格，内部允许继续拆分模块和替换后端，但不能破坏已确认的兼容性基线。

## 设计目标

XGE 是 2D 为主、允许轻量 2.5D 的游戏引擎。渲染管线必须满足以下目标：

- 默认路径适合大量 2D sprite、shape、text、UI 绘制。
- 高级路径允许自定义 shader、material、mesh、render target 和 2.5D 顶点。
- 公共 API 使用 Core Profile 风格，不依赖固定管线。
- 后端不能绑定桌面 OpenGL Core Profile 作为唯一实现，必须能映射到 GL 3.3、GLES 3、WebGL2 和小程序 WebGL2 风格接口。
- 普通绘制 API 应该可以进入命令队列，避免业务线程直接操作 GL。
- 第一阶段可先使用单线程即时 GL 实现验证功能，但命令队列和 render thread 是目标结构。

## 后端分层

渲染后端分为三层：

1. 平台层：创建窗口、context、swapchain 或等效对象。
2. GL 抽象层：封装 OpenGL/GLES/WebGL 差异，包括函数声明、常量、shader prelude、FBO 和 buffer 调用差异。
3. XGE 渲染层：实现 texture、buffer、shader、material、mesh、sprite batch、shape、text、render target 和 frame stats。

当前默认平台层使用 Sokol 负责窗口和 context。离屏、小程序、iOS/macOS 等目标后续可以替换平台层，但 XGE 渲染层不应该感知窗口库细节。

## 帧生命周期

游戏循环模式下，典型顺序为：

```c
xgeFrame();
// internal:
// poll input
// update scene
// begin frame
// draw scene
// flush commands
// present
```

APP/RMGUI 手动刷新模式下，典型顺序为：

```c
xgeBegin();
xgeXuiDraw(pRoot);
xgeEnd();
xgePresent();
```

约定：

- `xgeBegin()` 开启一次显式提交区间。
- `xgeEnd()` 会调用 `xgeFlush()` 并关闭提交区间。
- `xgeFlush()` 只负责把当前已提交命令执行到 GPU，不代表 swap/present。
- `xgePresent()` 用于手动模式的最终提交，当前会 flush 并清理 dirty rect。
- 自动游戏循环必须在每帧开始重置 transient render command 状态，避免上一帧未结束的 begin 状态污染下一帧。

## 命令提交模型

目标模型：

- `xgeDraw*`、`xgeShape*`、`xgeText*` 等公开绘制 API 写入 render command queue。
- command queue 支持主线程提交，后续扩展为多线程提交。
- GPU 对象创建、销毁和上传可由后台线程生成请求，但最终必须在拥有 GL context 的 render thread 执行。
- `xgeFlush()` 将队列按 pass、target、material、texture、blend、shader 等状态执行。

第一阶段当前实现：

- 部分绘制 API 仍直接调用 GL。
- `xgeBegin/xgeEnd/xgeFlush/xgePresent` 已作为公开生命周期入口存在。
- 已有 `xgeSpriteBatch` 用于手动批处理 sprite。
- 后续必须把即时 GL 路径逐步迁移到命令队列，迁移时保持公开 API 不变。

命令类型建议：

- clear
- draw sprite
- draw shape
- draw text
- draw mesh
- set viewport
- set camera
- begin pass
- end pass
- set scissor
- custom material draw
- upload texture/buffer
- destroy GPU resource

## 批处理策略

默认策略：

- 保持提交顺序，保证游戏和 UI 绘制结果稳定。
- 同一 pass 内可以合并连续且状态兼容的 draw。
- UI/XUI 默认不做跨命令重排。
- 允许高级 API 在特定 pass 开启 texture/material 排序，但这必须由用户显式选择。

批处理 key：

- render target
- viewport/scissor
- shader/material
- texture
- blend mode
- sampler
- vertex format
- primitive type

Sprite batch 顶点格式使用 `xge_vertex_t` 的子集或兼容布局：

```c
typedef struct xge_vertex_t {
	float fX;
	float fY;
	float fZ;
	float fW;
	float fU;
	float fV;
	uint32_t iColor;
} xge_vertex_t;
```

2D sprite 默认使用 `fZ = 0`、`fW = 1`。2.5D 和透视扭曲路径可以使用非零 `fZ` 或自定义 `fW`。

## 坐标与变换

XGE 支持两类坐标 API：

- 逻辑坐标：浮点，受 camera、viewport、逻辑分辨率映射影响。
- 像素坐标：整数或像素语义，适合 UI、调试图形和精确贴图。

默认 Y 轴向下。camera transform 与 viewport 相互独立，默认重合。

渲染时推荐统一转为 clip space：

1. local/object space
2. world/camera space
3. viewport/logical mapping
4. clip space

像素 API 可以视为逻辑 API 的包装，但必须绕开不必要的复杂矩阵路径，保证 UI 和 debug 绘制开销可控。

## RenderTarget 与 RenderPass

RenderTarget 用于：

- 离屏渲染。
- 后处理。
- 截图/readback。
- 小程序或无窗口环境下的输出目标。
- XUI 或 RMGUI 局部刷新缓存。

当前公开 API：

```c
int xgeRenderTargetWindow(xge_render_target pTarget);
int xgeRenderTargetCreate(xge_render_target pTarget, int iWidth, int iHeight);
int xgeRenderTargetResize(xge_render_target pTarget, int iWidth, int iHeight);
int xgeRenderTargetReadPixels(xge_render_target pTarget, void* pPixels, int iStride);
xge_texture xgeRenderTargetTexture(xge_render_target pTarget);
void xgeRenderTargetFree(xge_render_target pTarget);
```

约定：

- Window target 是默认输出目标。
- Texture target 内部使用 FBO + RGBA8 texture。
- `xgeRenderTargetReadPixels()` 是慢路径，不能用于每帧高频逻辑。
- RenderPass 第一阶段可以保持轻量结构，先覆盖 target、viewport、clear color、clear flags。
- 后续扩展 depth/stencil 时，不能影响纯 2D 默认路径。

## Shader 与 Material

Material 是普通用户高级绘制入口，负责把 shader、texture、color、blend 和 pipeline state 组合起来。

当前公开 API：

```c
void xgeMaterialInit(xge_material pMaterial);
void xgeMaterialFree(xge_material pMaterial);
void xgeMaterialSetShader(xge_material pMaterial, xge_shader pShader);
void xgeMaterialSetTexture(xge_material pMaterial, xge_texture pTexture);
void xgeMaterialSetColor(xge_material pMaterial, uint32_t iColor);
void xgeMaterialSetBlend(xge_material pMaterial, int iBlend);
void xgeMaterialDraw(xge_material pMaterial, const xge_draw_t* pDraw);
```

Shader 是更底层的高级接口：

- 内置 shader 由 XGE 生成或维护多平台版本。
- 用户 shader 第一阶段要求用户提供目标平台可用 GLSL。
- 后续可以增加 shader prelude，统一 attribute、varying、precision、texture sampling 宏。
- 不在第一阶段引入复杂 shader language/transpiler。

内置 shader 最少需要：

- sprite textured shader
- shape color shader
- text atlas shader
- material effect shader
- 2.5D mesh shader

## Blend 模式

第一阶段保留以下模式：

- `XGE_BLEND_NONE`
- `XGE_BLEND_ALPHA`
- `XGE_BLEND_ADD`
- `XGE_BLEND_MULTIPLY`
- `XGE_BLEND_SCREEN`
- `XGE_BLEND_CUSTOM`

默认纹理和颜色路径使用 premultiplied alpha。贴图加载阶段应尽量完成预乘，渲染阶段避免反复转换。

Bitmap era 的 CPU blend、GDI blend、逐像素滤镜不进入渲染核心。灰度、shade、distortion、mask 等效果应走 shader/material。

## Shape 与 Text

Shape 是调试、UI 和基础图形能力的一部分，不应该依赖 CPU bitmap blend。

当前 shape API 包含点、线、矩形、圆、圆弧、三角形、多边形等。后续迁移到命令队列时：

- 小 shape 可以进入 colored vertex batch。
- 圆和圆弧第一阶段可 CPU 生成顶点。
- 复杂矢量路径不作为 MVP 必须项。

Text 第一阶段由 stb_truetype 和 XRF 支持：

- TTF 负责动态栅格化和 glyph atlas。
- XRF 作为快速加载、快速渲染的点阵字格式，也可作为 TTF 缓存后端。
- 文字编码统一 UTF-8，第一阶段支持 UCS2 范围中文，不支持 emoji fallback。

## 2.5D 路径

2.5D 是 2D 引擎中的高级渲染能力，不代表 XGE 变成 3D 引擎。

允许用途：

- 四边形透视扭曲。
- 等距/伪 3D 层级。
- 卡牌翻转、UI 翻页。
- 地图或 sprite 的深度排序。
- 简单 mesh 特效。

约束：

- 不提供完整 3D 场景系统。
- 不提供骨骼动画、PBR、复杂光照、3D 资源管线。
- `xgeDrawQuad3D()`、`xgeMesh*()`、custom material 是 2.5D 的主要入口。

## 调试与统计

Debug 模式提供：

- GL error check。
- GPU caps dump。
- frame stats。
- draw call count。
- batch count。
- dirty rect count。
- texture/font/audio 资源统计。

发布版不强制保留这些检查，避免运行时开销。

当前公开 API：

```c
xge_frame_stats_t xgeFrameStatsGet(void);
int xgeDebugGetStats(xge_debug_stats_t* pStats);
int xgeDebugDumpCaps(char* sBuffer, int iSize);
```

## 迁移路线

渲染管线后续按以下顺序推进：

1. 完成 command queue 数据结构和单线程 flush。
2. 将 `xgeDraw*` 迁移到 command queue。
3. 将 shape/text/material draw 迁移到 command queue。
4. 增加 batch 合并统计和测试。
5. 增加 GPU upload/destroy queue。
6. 增加 render thread 执行模型。
7. 完善 RenderPass API。
8. 补齐 GLES3/WebGL2 shader prelude。
9. 为小程序和离屏后端补充后端适配层。

迁移过程中公开 API 不应频繁变化。若发现现有 API 无法支撑命令队列或多后端，必须先更新 spec 和设计文档，再修改代码。
