# XGE 专用渲染线程设计

本文档用于约束后续实现“拥有 GL context 的专用 render thread”。当前代码已经有 `xgeRenderThreadSet` 和 `xgeRenderThreadCapsGet`。基础路径仍支持同步等待的 worker drain；EGL/offscreen 路径已经增加 `xgeRenderThreadEGLSet`，可让 render thread 在线程内创建并持有 EGL/GLES3 context 后执行同步 flush。

## 当前边界

当前能力：

- 提交侧命令队列具备基本线程安全。
- `xgeFlush` 可以在无活动 Sokol 窗口 context 时使用 xrt worker drain。
- `xgeRenderThreadEGLSet` 可以配置 EGL pbuffer/surfaceless/native-window desc。
- EGL 配置成功并启用 `xgeRenderThreadSet(1)` 后，worker 会在线程内创建 EGL context、加载 GLES3 函数、flush 纹理上传队列和绘制命令，然后释放 context。
- `xgeRenderThreadCapsGet` 会明确报告：
	- `bWorkerDrain = 1`
	- `bGLContextOwned = 1`，当 EGL render thread 配置并启用成功时
	- `bGLContextOwned = 0`，当只使用普通 worker drain 时
	- `bAsyncFlush = 0`
	- `bCanUseWithCurrentContext = 0`，当 Sokol 窗口 context 正在运行时

当前不允许：

- 后台线程直接调用窗口线程创建的 GL context。
- 在 Sokol 默认回调生命周期外强行迁移 context。
- GPU resource create/destroy 在多个线程无序发生。
- `xgePresent` 与后台 GL flush 并发执行。
- 把 Sokol 窗口 context 交给 render thread；Sokol 路径仍保持主线程 GL。
- 把当前 EGL render thread 视为最终异步渲染模型；当前实现仍是同步 flush，且 context 生命周期是每次 worker flush 创建/释放。

## 目标模型

完整专用渲染线程应采用单一 context owner 模型：

- 平台层创建窗口和 GL/EGL/WebGL 等 context。
- context 所有权转交给 render thread，或由 render thread 创建 context。
- 所有 GL/GLES/WebGL 等价调用只在 render thread 执行。
- 游戏线程、资源线程、XUI 线程只提交命令，不直接触碰 GL。
- `xgeFlush` 提交当前命令批。
- `xgePresent` 提交 present 请求，并根据运行模式选择同步等待或异步排队。

## 线程角色

主线程：

- 负责平台消息循环。
- 负责用户 scene update、输入派发、XUI 事件派发。
- 负责构建 render command。
- 不直接调用 GL。

资源线程：

- 负责文件 IO、解码、压缩包读取。
- 生成 CPU 侧 image/font/sound 数据。
- 将 GPU upload/destroy 请求提交到 render queue。

渲染线程：

- 拥有 GL context。
- 执行 clear、pass、draw、upload、destroy、readback、present。
- 维护 frame stats、draw call、batch count。
- 接收 context lost/restored 或 resize 请求。

## 命令队列要求

完整实现前，命令队列需要扩展到以下类别：

- frame begin/end
- pass begin/end
- clear
- viewport/scissor
- blend/depth state
- draw sprite
- draw shape
- draw text
- draw mesh
- material draw
- texture/buffer upload
- texture/buffer/shader/material destroy
- render target create/resize/destroy
- readback request
- present request
- fence/sync request

所有命令必须满足：

- 数据自包含，不能引用栈内存。
- 引用 GPU/资源对象时必须增加引用计数。
- 命令执行完成后统一释放引用。
- 错误码可回传到提交侧或 debug stats。

## 同步策略

第一版完整 render thread 建议采用同步 flush：

- `xgeEnd` 调用 `xgeFlush`。
- `xgeFlush` 提交命令批并等待 render thread 执行完成。
- `xgePresent` 提交 present 并等待完成。

这样可以先保证行为稳定，再扩展异步：

- 后续可增加双缓冲命令队列。
- 后续可增加 frame fence。
- APP/manual 模式可以继续使用同步刷新，避免 UI 状态落后一帧。
- 游戏循环模式可选择异步提交，减少主线程等待。

## 平台路径

Sokol 窗口后端：

- 需要确认 Sokol 是否允许目标平台上的 context 迁移或外部 context 接管。
- 如果 Sokol 不适合 context 迁移，则 Sokol 路径保持主线程 GL，专用 render thread 只用于非 Sokol 或自建平台后端。
- Windows/Linux/macOS 可能支持不同策略，不能假设一致。

EGL/offscreen/board Linux：

- 更适合第一批完整 render thread 验证。
- render thread 可以创建并持有 EGL pbuffer/surfaceless context。
- 窗口显示链路后续再接 DRM/KMS/GBM 或 native window。

WebGL/小程序：

- 不作为第一版完整 render thread 目标。
- JS/WebGL 环境通常要求渲染在主线程或特定 worker/canvas 模型中执行。
- 只通过能力报告暴露实际可用边界。

## 实施顺序

1. 扩展 render command 类型，覆盖 texture upload/destroy 和 render target create/destroy。
2. 把现有即时 GL 资源创建路径改为可选提交到 GPU queue。
3. 为 EGL/offscreen 增加 render thread 创建 context 的内部入口。已完成：`xgeRenderThreadEGLSet` + worker 内 EGL context 创建。
4. 实现同步 `xgeFlush` fence。已完成：当前 worker flush 仍同步等待。
5. 实现同步 `xgePresent` command。
6. 在 `xgeRenderThreadCapsGet` 中按平台报告 `bGLContextOwned = 1`。已完成：EGL render thread 配置并启用后报告为 1。
7. 添加 offscreen render thread 自动测试。
8. 在真实板卡上做 EGL render thread 冒烟测试。
9. 再评估 Sokol 窗口后端是否接入完整 render thread。

## spec 关闭条件

`XGE_V2_SPEC.md` 中“实现拥有 GL context 的专用 render thread”只有在以下条件同时满足时才能关闭：

- 至少一个平台后端由 render thread 创建或持有 GL/EGL context。
- 所有 GPU create/upload/destroy/draw/present 都通过 render thread 执行。
- `xgeRenderThreadCapsGet` 能正确报告 `bGLContextOwned = 1`。
- 有自动测试覆盖无窗口/offscreen 同步 flush。
- 有人工冒烟测试验证真实窗口或真实 EGL 设备路径。

## 当前最终线程模型约定

当前阶段采用“单一 GL context owner”作为最终方向，但实现分阶段关闭：

- Sokol 窗口路径暂不强制迁移 context，保持主线程 GL 或平台默认模型。
- EGL/offscreen/board Linux 是专用 render thread 的优先验证路径。
- 资源线程只做 IO、解码和 CPU 数据准备。
- GPU create/upload/destroy/draw/present 最终都必须进入 render thread 或当前 context owner 的命令队列。
- `xgeFlush` 第一版保持同步 fence，先保证行为确定。
- 异步 present、双缓冲命令队列和 frame fence 属于后续优化。

文档、代码和测试中不得把当前 worker drain 误写成最终异步渲染模型。当前 worker drain 是过渡实现；真正关闭任务必须满足上面的 spec 关闭条件。
