# XGE 平台支持矩阵

本文档记录 XGE V2 第一版平台兼容目标、图形 API 基线、窗口/上下文方案和当前实现状态。

## 图形 API 基线

第一版图形能力基线：

- Desktop OpenGL 3.3 Core Profile
- OpenGL ES 3.0
- WebGL 2.0

XGE 允许使用 Core Profile 风格的现代渲染方式，但不把桌面 Core Profile API 绑定为唯一后端。内部应保持 GL Core / GLES3 / WebGL2 三者可映射。

## 平台矩阵

| 目标环境 | 窗口/上下文方案 | 图形 API | 第一版优先级 | 当前状态 |
| --- | --- | --- | --- | --- |
| Windows | Sokol | OpenGL 3.3 Core | 高 | 当前主要开发与构建验证平台 |
| Linux 桌面 X11 | Sokol | OpenGL 3.3 Core 或 GLES3 | 高 | 已有 X11 风格构建脚本和目标能力报告，需 Linux 实机验证 |
| Linux 桌面 Wayland | 升级 Sokol 或独立 Wayland/EGL 后端 | OpenGL 3.3 Core 或 GLES3 | 中 | 当前 vendored Sokol 未包含原生 Wayland，已保留能力检测/fallback 设计 |
| macOS | Sokol | OpenGL Core | 低 | 已有 macOS OpenGL MVP 构建脚本，需 macOS 实机验证 |
| Android | Sokol NativeActivity | OpenGL ES 3.0+ | 高 | 已有 NativeActivity/CMake、Gradle APK scaffold 和 adb 启动脚本，需 NDK/Gradle 构建和设备验证 |
| iOS | Sokol | OpenGL ES 3.0 | 低 | 已有 iOS simulator app scaffold 和 simctl 启动脚本，需 Xcode 构建和设备/模拟器验证；暂不接 Metal |
| Web/Emscripten | Sokol/Emscripten | WebGL 2.0 | 高 | 已有 Web 构建脚本、serve 脚本和 HTML shell scaffold，需 Emscripten/browser 验证 |
| 板卡 Linux | 自研 EGL 后端 | OpenGL ES 3.0+ | 高 | 已有 EGL 能力报告、pbuffer 路径和板卡 pbuffer 冒烟脚本，需目标设备验证 |
| 小程序 | 独立小程序后端 | WebGL 2.0 Canvas | 高 | 已有 JS bridge、WebGL2 Canvas 骨架和 hello scaffold，需真实小程序环境验证 |
| 离屏渲染 | EGL pbuffer/surfaceless | OpenGL ES 3.0+ | 高 | 已有 RenderTarget、EGL pbuffer init 和 CPU fallback 标记；surfaceless 未完成 |

## 当前已验证范围

当前自动构建和测试主要覆盖：

- Windows 本地构建。
- `xge.c + src/` 聚合编译。
- DLL 输出。
- EXE 示例输出。
- 纯 C 单元/回归测试。
- Sokol 窗口路径的代码编译。
- macOS/Android/iOS 构建入口已提供，但未在目标平台验证。
- Web/Emscripten 已提供构建脚本和浏览器 shell，但未在 Emscripten/browser 环境验证。
- 平台后端工具链检查脚本已提供，用于快速判断当前机器是否具备目标平台构建条件。
- 小程序 scaffold 文件生成。
- EGL/offscreen 的接口、能力报告和无 EGL fallback 状态。

未完成实机验证的平台不得在文档或发布说明中标为“已支持”。

平台验证结果统一记录在 `docs/平台后端验证结果.md`。没有记录环境、脚本、观察结果和关键日志的平台，不得关闭对应 spec 项。

## Sokol 覆盖范围

Sokol 是第一版默认窗口与事件后端，主要覆盖：

- Windows
- Linux X11
- Linux Wayland
- macOS
- Android
- iOS
- Web/Emscripten

XGE 不把 Sokol 作为唯一平台后端。板卡 Linux、离屏渲染和小程序需要独立后端。

## EGL 覆盖范围

EGL 后端用于：

- 板卡 Linux
- 离屏渲染
- pbuffer context
- surfaceless context

当前实现已经提供 `xgeEGLCapsGet`，用于区分当前构建是否编译了 EGL 路径，以及 pbuffer、surfaceless、native window、board Linux 这些能力是否可用。未启用 `XGE_HAS_EGL` 的构建会报告 `egl-unavailable`，这表示只有接口和 CPU-side render target 慢路径可用，不代表已经创建真实 EGL context。

`xge_egl_caps_t` 和 `xge_egl_context_t` 会记录 `iLastError` 与 `sLastStage`。在板卡上如果 EGL 初始化失败，应优先查看 `sLastStage`，它会指出失败阶段，例如 `eglGetDisplay`、`eglInitialize`、`eglChooseConfig`、`eglCreateContext`、`eglMakeCurrent` 或 `EGL_KHR_surfaceless_context`。

EGL 能力位按实际实现拆分：`XGE_HAS_EGL_PBUFFER`、`XGE_HAS_EGL_SURFACELESS`、`XGE_HAS_EGL_NATIVE_WINDOW`、`XGE_HAS_EGL_BOARD_LINUX`。单独定义 `XGE_HAS_EGL` 只表示编译了 EGL 模块，不自动承诺具体 surface 路径可用。

优先策略：

1. 平台支持 surfaceless 时优先使用 EGL surfaceless。
2. 不支持时使用 EGL pbuffer。
3. 板卡显示输出后续按目标系统选择 DRM/KMS/GBM、Wayland 或 X11/EGL。

## 小程序覆盖范围

小程序后端不走 Sokol，采用独立桥接。

第一版目标：

- WebGL2 Canvas context。
- Touch event bridge。
- 基础 keyboard/text event bridge，视平台能力启用。
- Resource loading bridge。
- Game loop bridge。
- 音频 bridge 策略或 miniaudio 可行性验证。

小程序是第一版核心需求，但与桌面/移动原生后端解耦实现。

## macOS 与 iOS 策略

macOS/iOS 是最低优先级兼容目标。

第一版暂时只走 OpenGL/OpenGL ES。Metal 后端只保留架构可能性，不进入第一版实现范围。

原因：

- 平台限制多。
- OpenGL/OpenGL ES 在 Apple 平台处于维护/兼容状态。
- 第一版目标是先完成统一 2D 引擎能力和核心跨平台路径。

## 板卡 Linux 假设

板卡 Linux 不追求覆盖远古设备。第一版可假设设备为近几年板卡，例如 RK3566 级别。

默认期望：

- 支持 OpenGL ES 3.0 或更高。
- 支持 EGL。
- 有可用 GPU 驱动。
- 显示、输入、音频能力按实际板卡发行版做适配。

## 平台能力报告

后续需要实现平台能力报告，至少包含：

- 平台后端名称。
- 图形后端名称。
- GL/GLES/WebGL version。
- GLSL version。
- 最大纹理尺寸。
- RenderTarget 支持情况。
- Offscreen context 支持情况。
- Touch/gamepad/text input 支持情况。

当前已有 `xgePlatformCapsGet()` 提供平台能力报告，`xgeDebugDumpCaps()` 会同时输出平台后端、图形后端、窗口、离屏、RenderTarget、输入和音频等基础能力。该报告表示当前后端的接口能力与降级策略，不等同于目标平台已完成实机验证。

`xgePlatformRuntimeGet()` 提供当前运行态快照，用于冒烟测试记录：

- `bRunning`：当前平台窗口/上下文是否处于运行态。
- `iWindowWidth/iWindowHeight`：窗口逻辑尺寸。
- `iFramebufferWidth/iFramebufferHeight`：实际 framebuffer 像素尺寸。
- `fDpiScale`：framebuffer 与窗口逻辑尺寸的比例。
- `iKeyEventCount/iTextEventCount/iMouseEventCount/iTouchEventCount/iGamepadEventCount/iResizeEventCount/iQuitEventCount`：平台事件计数。

`examples/platform_smoke` 会定期打印 runtime 快照。目标平台人工验证时，应观察窗口尺寸、framebuffer 尺寸、DPI scale 和输入事件计数是否随操作变化。

Sokol 后端会根据编译目标报告 `sSokolTargetName` 和对应能力位：Windows、Linux X11、Linux Wayland、macOS、Android、iOS、Web/Emscripten。报告中也会包含 Sokol 编译图形后端能力位：`bSokolGLCore`、`bSokolGLES3`、`bSokolD3D11`、`bSokolMetal`、`bSokolDummy`。这些字段用于诊断当前二进制到底以哪种 Sokol 图形后端编译，尤其用于 Android/iOS/Web/Linux 目标机冒烟时快速确认没有误用桌面 GL 或 dummy backend。

输入能力按后端语义拆开报告：

- `bTouch` 表示当前目标预期有触摸输入。
- `bMouse` 表示当前目标预期有鼠标/指针输入。
- `bKeyboard` 表示可接收按键事件。
- `bTextInput` 表示可接收文本输入/系统 IME 结果。
- `bGamepad` 表示后端保留 gamepad 事件路径。
- `bHighDPI` 表示后端保留高 DPI / device scale 路径。

这些能力位是接口与编译路径报告，不替代人工输入验证。触摸、IME、gamepad 和高 DPI 的真实行为仍要按 `docs/人工冒烟测试流程.md` 在目标平台确认。

Sokol 非 Windows 平台的具体验证步骤见 `docs/Sokol跨平台冒烟测试计划.md`。

当前 vendored Sokol 的 Linux 桌面实现基于 X11，未包含原生 Wayland 后端。具体边界见 `docs/Sokol后端边界说明.md`。

## 发布标记规则

发布说明中平台状态使用以下等级：

- `已验证`：在目标平台完成构建和冒烟测试。
- `可编译`：脚本或工程可构建，但未完成运行验证。
- `骨架`：接口或后端骨架已存在，但不可视为可用。
- `计划`：设计已确认但尚未实现。
- `低优先级`：保留兼容目标，但不阻塞第一版。

不得把 `计划` 或 `骨架` 写成 `已支持`。
