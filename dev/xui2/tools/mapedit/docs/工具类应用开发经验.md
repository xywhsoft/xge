# XUI 工具类应用开发经验

本文记录 MapEdit 开发过程中已经验证过的 XUI 工具类应用经验。目标是把 XUI 从“能做游戏 UI”扩展到“能长期维护复杂工具软件”的开发方式。

## 程序结构

工具类程序应优先使用代码搭建页面，而不是把整个界面写成一个 page 文件。page/xson 更适合描述固定页面，工具软件更需要模块化代码、跨窗口状态同步、命令路由和按需扩展。

推荐结构：

- `src/core`：应用入口、菜单、状态栏、工作区切换、配置加载、布局保存。
- `src/workspace`：通用 DockPanel 窗口创建、占位页、布局加载保存封装。
- `src/workspaces/<workspace>`：具体工作区。
- `src/workspaces/<workspace>/tabs`：每个 tab 页一个独立源文件。
- `map_sdk`：地图数据、图块集、渲染等可移植模块，供编辑器和运行时共用。
- `option`：用户和默认配置、布局文件。
- `release`：最终运行目录，exe、dll、res、option 都复制到这里。

每个 tab 页独立文件维护，路径按“工作区 / tab 页名称”拆分，避免后期几十个页面集中在一个大文件里。

## 布局原则

工具类应用必须避免绝对坐标。主窗口推荐使用纵向布局：

1. 菜单栏。
2. 工具栏或工作区切换栏。
3. DockPanel 主工作区，占据剩余空间。
4. 状态栏。

客户区如果需要多个互斥工作区，使用 `XGE_XUI_LAYOUT_STACK`。每个工作区各自持有一个独立 DockPanel 容器，不能混用 DockWindow，避免图块编辑器窗口被加载到地图编辑器里。

左右两侧 Dock 区域适合固定像素宽度，中间区域占据剩余空间。默认窗口尺寸目前适合使用 `1400x900`，方便容纳三栏 DockPanel 和多 tab 页。

## DockPanel 布局持久化

布局保存口径是 DockPanel 布局，不是 page 保存。DockPanel 的布局文件应按工作区拆分：

- `layout_tileset.xson`
- `layout_map.xson`
- `default_layout_tileset.xson`
- `default_layout_map.xson`

布局文件放在 `option` 目录。运行目录下也应有 `release/option`，程序实际读取运行目录旁边的 option 文件。

加载顺序：

1. 优先加载用户布局。
2. 用户布局不存在或失败时加载默认布局。
3. 默认布局也失败时使用内置布局。

程序退出时自动保存当前 DockPanel 布局。不要把不同工作区保存进同一个布局文件。

## 工作区切换

复杂工具不要继续在 DockPanel 里叠加一层工作区 tabs。工作区切换更适合放在顶部工具栏，类似 RPG Maker 数据库模式。

经验：

- 工具栏按钮可以作为 tabs 使用。
- 按钮需要单选互斥。
- 反复点击当前选中按钮不切换页面。
- 选中态属于业务状态，hover/active 属于控件瞬态状态，两者不要互相清空。

已踩坑：工作区按钮点击时如果手动把 `iState` / `iVisualState` 清成 normal，会和按钮内部 hover/active/capture 状态冲突，产生闪烁。正确做法是只更新 checked/selected 视觉，保留当前 hover/active/focus/disabled 瞬态状态。

对于作为 tabs 使用的工具栏按钮，active 视觉不宜和 hover/selected 差异过大。否则在事件或刷新边界上容易出现按下态和悬停态的可见跳变。

## 工具栏样式

工具类应用的工具栏默认应低调：

- 工具栏默认不画边框，避免和菜单、客户区边框叠成 2px。
- 按钮默认无边框、无背景。
- hover 时显示淡蓝背景和深蓝边框。
- active 时显示更明确的按下反馈。
- 如果按钮作为工作区 tabs 使用，active 可与 hover/selected 保持接近，避免闪烁。

图标建议 24x24。文字在图标下方时，按钮尺寸要稳定，避免 hover/active 改变布局。

## 按需刷新

XUI 工具应用不应该固定每帧重绘。游戏适合持续 frame loop，工具适合按需刷新。

当前已引入 `XGE_INIT_ON_DEMAND`：

- 初始启动请求一帧。
- 输入、鼠标、触摸、resize、quit 等明确影响窗口内容的事件请求一帧。
- `xgeXuiRefreshRequest()` 会调用 `xgeRenderRequest()`。
- 没有刷新请求时，Sokol frame callback 直接返回，不做 `glClear`，不跑 scene update/draw。

注意：这不是严格的脏矩形局部重绘。它解决的是“空闲不重绘”。真正的局部矩形重绘还需要 scissor、分层缓存、遮挡处理、窗口重叠处理和外部 surface invalidation。

## 控件状态与刷新

按需刷新后，任何应用状态变化都必须显式标记相关控件重绘。

典型例子：素材预览 hover 格子。

错误做法：

```c
pSource->iHoverCol = iCol;
pSource->iHoverRow = iRow;
```

这样只改了数据，没有请求绘制。结果是鼠标移动后要等别的 UI 触发刷新，hover 才显示出来。

正确做法：

```c
if (oldCol != pSource->iHoverCol || oldRow != pSource->iHoverRow) {
    xgeXuiWidgetMarkPaint(pWidget);
}
```

只在状态实际变化时 `MarkPaint`，避免鼠标在同一个格子内移动也频繁重绘。

通用规则：

- 改变视觉数据后调用 `xgeXuiWidgetMarkPaint()`。
- 改变尺寸、布局参数后调用 `xgeXuiWidgetMarkLayout()`。
- 批量布局修改时使用 layout batch。
- 不要依赖“下一帧总会刷新”这种游戏式假设。

## 状态栏

状态栏适合显示短字段，不适合塞长句。字段宽度要预留足够空间，避免中文显示被截断。

推荐字段：

- 当前工作区。
- 地图尺寸。
- 鼠标所在坐标。
- 当前图块集、图块 ID、图层。
- 通行、区域、事件等位置附加信息。

状态栏内容后续会随着业务完善，但框架阶段也应该放占位字段，便于提前观察布局稳定性。

## 字体

工具类 UI 当前优先使用宋体 12px XRF 字库。这个尺寸在当前渲染路径下更清晰，适合菜单、Dock tab、状态栏、普通标签。

经验：

- 统一所有 UI 字体，不要菜单、tabs、状态栏各用不同字体。
- 需要时再增加宋体 16px，用于局部标题或重要字段。
- 字体资源放在 `release/res/fonts`。
- 工具类 UI 不追求游戏式平滑字体，清晰优先。

## 资源与发布目录

MapEdit 的运行目录是 `tools/mapedit/release`。发布时应保证：

- `xge_mapedit.exe`
- `xge.dll`
- `res/*.png`
- `res/fonts/*.xrf`
- `option/*.xson`

配置文件应和 exe 同目录下的 option/res 组合使用。路径拼接优先使用 `xrtPathJoin`，基准为 `xCore.AppPath`。

配置文件用于描述地图编辑器生成资源的规格，例如：

- 图块宽高。
- 图层数量。
- 每行图块数量。
- 自定义数据通道数量与通道定义。

这些规格会影响素材预览区域大小、图块 ID 计算、图块集打标和后续地图数据格式。

## 图标

窗口图标和 DLL 图标分开：

- `xx_exe.ico` 放入地图编辑器 exe，用于程序文件和窗口展示。
- `xx_dll.ico` 放入 DLL，用于替换 sokol 默认窗口图标。

Windows 常用展示尺寸：

- 标题栏通常使用 16x16。
- 任务栏常用 32x32。
- 资源管理器可能使用更大尺寸。

如果只面向工具窗口和任务栏，图标资源不需要无限堆叠大尺寸，但保留常用多尺寸 ico 更稳。

## 编译验证

开发期间经常会遇到 `xge_mapedit.exe` 正在运行导致编译失败：

```text
cannot open output file ... xge_mapedit.exe: Permission denied
```

这是输出 exe 被锁住，不是源码错误。不要直接强杀用户正在操作的程序。可选处理：

- 提醒关闭运行中的 MapEdit 后重新编译。
- 或临时输出到 `xge_mapedit_check.exe` 做语法/链接验证。

正式构建命令：

```bat
cmd /c build_dll.bat
cmd /c tools\mapedit\build.bat
```

短启动验证：

```bat
cmd /c tools\mapedit\release\xge_mapedit.exe --frames 2
```

`--frames` 测试模式不启用按需刷新，避免自动测试没有后续帧而无法退出。

## 当前结论

XUI 可以同时支持游戏和工具，但两类应用的刷新模型不同：

- 游戏：持续帧循环，动画和输入每帧推进。
- 工具：按需刷新，状态变化显式标记控件脏。

工具类应用开发的关键不是“每帧少画一点”，而是建立清晰的 UI 状态、布局、重绘责任边界。只要控件状态不互相抢、应用状态变化能明确 `MarkPaint`，XUI 就能胜任复杂工具软件。
